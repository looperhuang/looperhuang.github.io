C# 面試問題紀錄

發表於 2025-11-19 分類於 面試
簡單記錄一下面試可能的問題或遇到的問題。

C# 中的值型別與參考型別差在哪？

  1. 值型別（Value Type）：存在 Stack，例如 int、bool、struct、enum
  2. 參考型別（Reference Type）：存在 Heap，例如 class、string、array
  • 傳遞時值型別是 複製、參考型別是 指向同一物件
  • string 雖然是參考型別，但不可變（immutable）

string 為什麼是不可變的？

  • 原因： 主要是為了安全和執行緒安全。
  • 安全： 確保字串的雜湊值固定，適合當作 Dictionary 的 Key。
  • 執行緒安全： 多個執行緒共用同一個字串時，因為它不能被改變，所以不會產生資料衝突

class 與 struct 的差別？什麼時候用 struct？

  • class (類別)： 參考類型，資料存在 Heap，傳遞的是記憶體地址。
  • struct (結構)： 值類型，資料存在 Stack，傳遞的是資料的完整拷貝。
  • 使用 struct： 當你的數據輕量、小巧（建議 16 位元組以下）、且不需要繼承時，適合用 struct 獲得效能優勢。

for、foreach 差別？哪個效能較好？

  • for： 基於索引或計數，用於精確控制。
  • foreach： 基於集合元素，用於遍歷。可讀性好，且不能在迴圈內修改集合。
  • 效能： 大多數情況下差異極小，優先選用可讀性高的 foreach。只有遍歷 Array 或 List 時，for 迴圈可能稍微快一點。

var 與 dynamic 差在哪？

  • var： 只是語法糖。類型在編譯時就確定了（靜態類型）。編譯後它就是一個確切的類型，無性能損失。
  • dynamic： 真正的動態類型。類型要到執行時才確定。失去了編譯時的錯誤檢查，有輕微性能開銷，用於特殊互操作。

using 區塊是做什麼的？

  • 作用： 確保非受管資源（如檔案連線、資料庫連線）能夠被自動、可靠地釋放。
  • 機制： using 區塊保證在它結束時，會呼叫物件的 Dispose() 方法，即使發生例外也一樣，避免資源洩漏。

    需要實作 IDisposable 介面

什麼是例外處理（Exception Handling）？try/catch/finally 的用途？

  • 例外處理： 讓程式在遇到非預期錯誤時不會直接崩潰，而是可以安全地處理或記錄錯誤。
  • try： 包圍可能發生錯誤的程式碼。
  • catch： 捕獲並執行錯誤發生後的補救邏輯。
  • finally： 無論有沒有錯誤，都一定會執行的清理程式碼（例如：關閉連線）。

如何避免 NullReferenceException？

  • 現代方法： 使用 Null 條件運算子 ?. 和 Null 合併運算子 ??。
  • 設計： 避免方法返回 null（例如：返回空集合而非 null）。
  • C# 8.0+： 啟用可空參考類型，讓編譯器在編譯時就警告你潛在的 null 風險。

什麼是 OOP？說出四大特性。

OOP 是一種程式設計典範，核心是將現實世界中的事物抽象化為物件 (Objects)。它透過讓這些物件彼此合作，來實現複雜的功能。

  1. 封裝 (Encapsulation)：

    • 作用： 把資料和方法綁在一起藏好，不讓外部隨意存取內部細節。
    • 目的： 隱藏複雜性，只暴露必要的介面。

  2. 繼承 (Inheritance)：

    • 作用： 允許新的類別（子類）繼承現有類別（父類）的屬性和方法。
    • 目的： 實現程式碼重用和建立 is-a 關係（例如：狗是一種動物）。

  3. 抽象化 (Abstraction)：

    • 作用： 專注於事物的本質和功能介面，忽略不重要的實作細節。
    • 目的： 讓使用者只看到他們需要的（例如：開車只看方向盤，不看引擎細節）。

  4. 多型 (Polymorphism)：

    • 作用： 允許不同的物件對同一個訊息或同一個方法做出不同的反應。
    • 目的： 程式碼具有彈性（例如：呼叫 Animal.MakeSound()，貓會「喵」，狗會「汪」）。

override、virtual、abstract 差別？

  • virtual (父類)： 我提供一個預設行為，子類可以選擇覆寫。
  • abstract (父類)： 我只定義一個方法規格，子類必須提供實現。
  • override (子類)： 我正在替換掉父類 virtual 或 abstract 方法的實作。

LINQ 的 Where()、Select() 有什麼作用？

  • Where() 的作用：篩選 (Filtering)
    Where 解決「哪些」： 決定哪些元素應該保留。
  • Select() 的作用：投影 (Projection)
    Select 解決「長怎樣」： 決定保留下來的元素應該長什麼樣（轉換成什麼）。

List 和 Array 差在什麼地方？

  • Array (陣列)： 長度固定，建立後不能改變。效能最好，適合數量不變的場景。
  • List (列表)： 長度可變，可以隨時新增/刪除。當空間不夠時會自動擴容，彈性高。

IEnumerable 是什麼？有什麼用途？

  • 是什麼： 是一個契約標籤，承諾這個集合是可以被遍歷的。
  • 用途： 它是 foreach 迴圈和 LINQ 查詢的基石。
  • 核心優勢： 實現延遲執行。它只在真正需要元素時才計算，適合處理大型資料，節省資源。

== 與 .Equals() 差在哪？

比較對象 == (運算子) .Equals() (方法)
預設行為 比較記憶體地址 (參考) 比較記憶體地址 (參考)
自定義 Class 預設比較地址，但可重載 預設比較地址，但可覆寫 (用來比較內容，實現邏輯相等)
String (特殊) 兩者都比較內容 兩者都比較內容

請解釋 C# 屬性（Property）的用途

  • 用途： 是一種受控的、安全的方式來存取類別的內部資料。
  • 核心： 透過 get 和 set 區塊，你可以在讀取或寫入資料時，加入驗證、計算或邏輯，防止外部直接亂改資料，實現封裝。

如何宣告一個泛型方法（Generic Method）？

  • 方法： 在方法名稱後面加上 ，T 就是佔位符。
  • 範例： public static void Swap(ref T a, ref T b)
  • 作用： 讓你寫一次邏輯，就能適用於 int、string 等多種資料類型。
public static T /* 1. 傳回類型可以是 T */
    MyGenericMethod<T> /* 2. 在方法名稱後用 <T> 宣告類型參數 */
    (T inputParameter, string message) /* 3. 參數也可以使用 T */
{
    // 方法內容
}

Overload 與 Override 差在哪？

  • Overload (重載)： 就像是同一個動詞 (方法名)，但可以有不同的用法 (參數)。例如，你有 Add(int a, int b) 和 Add(double a, double b) 兩個方法。
  • Override (覆寫)： 就像是繼承。子類替換掉父類的方法，實現多型。例如，Dog 覆寫了 Animal 的 MakeSound() 方法。

請說出 C# 常用的集合類別（Collection Types）有哪些？

類別 簡潔用途 特點
List 存放會變動的一串資料 最常用。可增長，內部基於陣列。
Array 存放固定數量的資料 效能最好，但長度不能變。
Dictionary<TKey, TValue> 用 Key 查 Value 查詢速度最快（平均 O(1)）。
HashSet 存放不重複的資料 查詢速度極快，不用 Key。

什麼是委派（Delegate）？它與事件（Event）差在哪？

  • 委派 (Delegate)： 就像一張「空白合約」，用來定義方法的規格（參數和返回值），讓你把方法當作參數傳遞。
  • 事件 (Event)： 是委派的一個安全郵箱。它只允許外部訂閱/退訂（+= / -=），但不允許外部隨意觸發或清空訂閱列表，確保系統穩定。
// 1. 宣告一個委派類型
public delegate void NotifyHandler(string message);

public class Publisher
{
    // 2. 宣告一個委派欄位 (這裡還不是事件)
    public NotifyHandler OnNotification;

    public void SendMessage(string msg)
    {
        // 3. 呼叫委派，所有註冊的方法都會執行
        OnNotification?.Invoke(msg);
    }
}

public class Subscriber
{
    public void Handle(Publisher p)
    {
        // 訂閱事件：允許
        p.OnNotification += MyMethod;

        // 外部呼叫：編譯錯誤！
        // p.OnNotification.Invoke("Hack!");

        // 外部清空：編譯錯誤！
        // p.OnNotification = null;
    }

    private void MyMethod(string message) { /* 處理邏輯 */ }
}

請說明 async/await 的運作方式

async/await 的核心是讓程式執行非同步操作時不阻塞主執行緒。

  1. async (標籤)： 告訴編譯器該方法內有非同步操作，需要特殊處理。方法必須返回 Task。
  2. await (暫停點)： 當程式執行到 await 時，會釋放當前的執行緒給其他任務使用，程式暫停。

    await 就像你對電腦說：「你去幫我處理這個網路下載任務（I/O 任務）。在你等待資料的這段時間，你不用傻等，你可以先去做其他事。等數據一回來，你再回來繼續執行我的後續程式碼！」

  3. 恢復： 等待的任務（如網路 I/O）完成後，系統會自動在一個新的執行緒上恢復 await 後續的程式碼執行。

    總結： 程式在等待 I/O 完成時，執行緒可以去做其他事情，實現高效利用資源。

Task 與 Thread 差在哪？

Task 和 Thread 的差異主要體現在層級和效率上。

  • Thread (執行緒)

    • 底層的勞力，直接對應作業系統的執行緒。
    • 成本高。啟動和銷毀資源消耗大，需要手動管理。現代 C# 很少直接使用。

  • Task (任務)

    • 高階的抽象，一個工作請求。
    • 效率高。它會自動交給 ThreadPool (執行緒池) 管理和重複利用底層的 Thread。

Task 是你發出的任務單，ThreadPool 是管理工人的經理，而 Thread 才是真正的工人。我們只需要發任務單，不需要自己管工人。

什麼是 Dependency Injection？為什麼要用？

控制反轉 (IoC) : 把「誰來決定建立物件」這件事，從自己手上交給別人（框架）處理。

  • 是什麼 : 類別不再自己建立所需的依賴物件，而是由外部（DI 容器）傳遞（注入）給它。

  • 為什麼用 : 實現高度解耦 (Decoupling)。類別只依賴抽象介面，而非具體實現。

  • 主要優勢 :

    1. 易於測試： 可以輕鬆替換為模擬物件 (Mock)。
    2. 高彈性： 替換底層服務（如資料庫）無需修改核心程式碼。

DI 中 Singleton / Scoped / Transient 差異？

  1. Singleton： 像公司的總裁，整個公司（應用程式）只有一個。無論誰找，都是找同一個人。
  2. Scoped： 像一個部門經理，在這次專案（Scope / HTTP Request）裡，大家找的都是同一個經理。但換到下一個專案，大家會創建和找新的經理。
  3. Transient： 像一份外賣訂單，每次你餓了（請求），DI 容器就給你做一份新的。

Entity Framework 的 Lazy Loading vs Eager Loading？

特性 Lazy Loading (延遲加載) Eager Loading (急切加載)
加載時機 需要時才加載。當你第一次存取導航屬性時（例如 order.Details），EF 才會發送查詢。 查詢主要實體時就加載。在最初的查詢中就同時抓取相關資料。
資料庫查詢 多次查詢。第一次查詢主實體，後續每次存取關聯物件都會發送一個新的查詢。 一次查詢。通常使用 JOIN 或多個查詢語句在資料庫端完成。
實現方式 需要導航屬性是 virtual (虛擬)。 需要使用 Include() 擴展方法。
效能影響 可能導致 N+1 問題 (性能殺手)。如果遍歷很多實體，會發出過多的小查詢。 效率較高，但可能一次載入過多不必要的資料，導致單次查詢變慢。
適用場景 關聯資料很少被存取；或只需要存取極少數實體的關聯資料。 確定需要使用關聯資料時；高效處理大量關聯實體。
  • Eager Loading (Include()): 推薦使用。在需要關聯數據時，明確告訴 EF 要抓取哪些數據，可以有效避免 N+1 查詢問題，提升效能。
  • Lazy Loading: 方便但危險。當你不小心在迴圈中存取關聯屬性時，會不知不覺中產生大量的資料庫往返，應謹慎使用。

LINQ 的 Deferred Execution 是什麼？

延遲執行是指 LINQ 查詢在程式碼中被定義時，並不會立即執行，而只是建立一個查詢定義 (Query Definition)。
實際的資料獲取和運算會被推遲到結果真正被需要的時候。

  • 資源效率 (Lazy Evaluation)：
    只有在你需要時才計算元素。如果你只需要查詢的前 5 筆資料，LINQ 就會停止計算後續的資料，節省運算時間和資源。
  • 優化查詢：
    對於 Entity Framework (LINQ to Entities)，延遲執行允許 EF 將多個 LINQ 操作合併並翻譯成一個優化過的 SQL 查詢，減少資料庫的往返次數。
  • 獲取最新數據：
    由於查詢是延遲執行的，每次執行時都會從底層資料源（例如資料庫）獲取最新的數據。

延遲執行就像是點餐不開火。

  1. 服務生（程式）先寫下你的訂單（查詢定義）
  2. 直到你確定要吃（開始遍歷或 ToList()）時
  3. 廚房（資料源）才開始製作（執行查詢）。

IEnumerable vs IQueryable 的差別？

特性 IEnumerable IQueryable
資料處理位置 本地端 (Local)：在應用程式的記憶體中處理資料。 遠端端 (Remote)：在資料庫伺服器端處理資料。
查詢執行 先拉後篩：從資料庫拉取所有符合條件的資料，到記憶體後再進行 Where 或 Select 篩選。 先篩後拉：將整個 LINQ 查詢翻譯成 SQL，讓資料庫伺服器執行篩選和運算，只拉取最終需要的結果。
適用範圍 所有集合：適用於任何實現 IEnumerable 的集合（如 List, Array）。 資料庫/遠端來源：通常由 ORM（如 Entity Framework）提供。
底層機制 基於 委派 和 迭代器。 基於 Expression Trees (表達式樹)。
性能影響 低效：如果資料庫中有 100 萬筆記錄，你可能會拉取 100 萬筆到應用程式的記憶體中。 高效：只拉取你需要的 10 筆記錄，節省網路帶寬和應用程式記憶體。
  1. IEnumerable： 適合處理已經在記憶體中的資料。
  2. IQueryable： 適合處理遠端資料庫的資料，它能將你的 C# 程式碼轉換成高效的 SQL 查詢。

核心記憶點： 永遠應該先使用 IQueryable 進行篩選和排序，在確定資料量變小後，最後再使用 ToList() 或 AsEnumerable() 轉換成 IEnumerable 進行本地處理。

請描述 RESTful API 的設計原則。

  1. 資源 (Resource-Based)：
    • 原則： 將網路上的所有事物視為資源，每個資源都透過唯一的 URL (URI) 來識別。
    • 實踐： URL 應該使用名詞來表示資源集合（例如 /users, /products/123），避免使用動詞。
  2. 統一介面 (Uniform Interface)：
    • 原則： 採用一組標準化的操作方法來處理資源。
    • 實踐： 使用標準的 HTTP 動詞對資源進行操作（CRUD）：
      • GET： 獲取（讀取）資源。
      • POST： 創建（新增）資源。
      • PUT / PATCH： 更新（修改）資源。
      • DELETE： 刪除資源。
  3. 無狀態性 (Statelessness)：
    • 原則： 伺服器不會在兩次請求之間儲存任何客戶端的狀態資訊。
    • 實踐： 客戶端發出的每一個請求都必須包含伺服器處理該請求所需的所有資訊（例如，認證資訊、Session ID 等）。
  4. 分層系統 (Layered System)：
    • 原則： 客戶端無法區分它是直接連線到最終的伺服器，還是中間有代理、負載平衡器或其他中介層。
    • 目的： 提高系統的可擴展性和安全性。

Garbage Collection 三代（Gen 0/1/2）如何運作？

.NET 的 GC 採用分代回收策略，目的是根據物件的存活時間將它們分配到不同的代（Generation），從而只檢查那些最有可能被回收的物件。

  1. Gen 0 (第一代：年輕的、短命的)

    • 物件： 剛創建的新物件。
    • 回收頻率： 最高，因為絕大多數物件（約 90% 以上）生命週期極短。
    • 運作： 空間滿時，GC 執行 Gen 0 回收。如果物件仍被引用（存活），則將其提升到 Gen 1。

  2. Gen 1 (第二代：中年的、過渡期)

    • 物件： 從 Gen 0 存活下來的物件。
    • 回收頻率： 較低。
    • 運作： Gen 0 的回收無法釋放足夠的記憶體時，GC 會執行 Gen 1 回收。存活下來的物件被提升到 Gen 2。Gen 1 充當一個緩衝區。

  3. Gen 2 (第三代：老年的、長命的)

    • 物件： 從 Gen 1 存活下來的物件，通常是長時間存活的物件（如靜態物件、單例服務）。
    • 回收頻率： 最低，且最昂貴。
    • 運作： 只有當 Gen 0 和 Gen 1 都無法釋放足夠記憶體時，才會執行 Gen 2 回收。GC 會遍歷整個堆積 (Heap)。

Web API 的 Middleware 運作順序？

Middleware 是一系列組件，它們在處理 HTTP 請求 (Request) 和生成 HTTP 響應 (Response) 的過程中，以一個管道 (Pipeline) 的方式依序執行。

  1. 洋蔥模型 (The Onion Model)
    Middleware 的執行順序遵循一個經典的洋蔥模型：
  • 請求 (Request)： 請求從最外層的 Middleware 依序進入到最內層（通常是應用程式的實際處理邏輯，如 Controller）。
  • 響應 (Response)： 響應從最內層的處理邏輯生成後，會反向穿出，依序經過每個 Middleware，直到返回給客戶端。
  1. 關鍵順序與控制
    Middleware 的順序由你在 Program.cs 檔案中呼叫 app.Use…() 方法的順序來決定。
  • 越早註冊，越早進入、越晚離開。
    • 外層 (先註冊)： 負責高階的任務，如異常處理、日誌記錄、安全性。
    • 內層 (後註冊)： 負責低階、具體的任務，如路由、CORS、身份驗證。

什麼是 Model Binding？（Web API）

Model Binding 是一種 ASP.NET Core 機制，它的作用是自動將 HTTP 請求中的數據，轉換並映射到方法的參數 (Model) 中。

參數類型 預設綁定來源
簡單類型 (Primitive types: int, string, bool 等) 預設從 路由、查詢字串、表單 中查找。
複雜類型 (Complex types: User 類別, ProductModel 等) 預設從 請求主體 (Request Body) 中查找，並使用格式化器 (Formatter) 來解析 JSON 或 XML。

Model Binding 就像一個翻譯官兼自動填表器，它將雜亂的 HTTP 請求數據，自動整理成乾淨、可用的 C# 物件，供你的程式碼直接使用

async/await 下如何避免 deadlock？

  • 全程使用 await
    • 永遠不要使用 .Wait(), .Result, 或 Task.GetAwaiter().GetResult() 來同步地等待一個 Task 完成。
    • 如果你的方法呼叫了一個 async 方法，那麼你的方法也應該是 async 的，並使用 await 來等待結果。
// 推薦：從頭到尾保持非同步
public async Task AsyncMethod()
{
    // ✅ 不阻塞，只是暫停並釋放執行緒
    await MyAsyncMethod();
}
  • 使用 ConfigureAwait(false)
    • 告訴 await 系統，不需要回到原來的同步上下文來恢復後續程式碼。
    • 在所有不關心 UI 或請求上下文的通用庫程式碼中，使用 ConfigureAwait(false)。
public async Task MyLibraryMethod()
{
    // ✅ 恢復時不需要回到原來的執行緒，可以找任何空閒執行緒
    await Task.Delay(100).ConfigureAwait(false);
}
  • 避免鎖定資源
    • 盡量不要在 await 關鍵字前後持有共享資源鎖（例如 lock 區塊）。因為 await 會釋放執行緒，可能導致鎖資源長期被佔用。

LINQ Join 的語法與用途？

var result = employees.Join(
    departments,                // 要連接的目標集合
    employee => employee.DeptId,  // 來源集合的 Key (外鍵)
    department => department.Id,  // 目標集合的 Key (主鍵)
    (employee, department) => new {  // 結果選擇器
        employee.Name,
        department.DeptName
    }
);
  • 資料整合： 將分散在不同列表或資料表中的資料，根據共同的 ID 關聯起來。
  • 等同 SQL INNER JOIN： 只有在兩個集合中都有匹配的 Key 時，才會產生結果。
  • 多對一關聯： 常用於將多個「子項」與一個「父項」對應起來。

如何建立 Custom Exception？為什麼要做？

  1. 繼承標準 Exception 類別

你的自定義例外類別必須直接或間接繼承自 System.Exception 或其衍生類別（最常見的是 System.ApplicationException 或直接繼承 System.Exception）。

  1. 實現標準建構函式
// 假設我們的應用程式不允許年齡小於 18 歲
public class AgeTooYoungException : Exception
{
    // 1. 無參數建構函式
    public AgeTooYoungException() : base("使用者年齡小於最小限制。") { }

    // 2. 帶訊息建構函式
    public AgeTooYoungException(string message) : base(message) { }

    // 3. 帶訊息和內部例外建構函式
    public AgeTooYoungException(string message, Exception innerException)
        : base(message, innerException) { }

    // (可選) 添加自定義屬性
    public int MinimumAge { get; set; } = 18;
}

用途 :
建立自定義例外的主要目的是為了讓程式碼更清晰、專業、易於偵錯，並提供特定情境的錯誤訊息。

lock、Monitor、SemaphoreSlim 差在哪？

用於控制多執行緒對共享資源的存取，但設計的目的和適用場景不同

機制 簡潔用途 核心特性 適用場景
lock 互斥鎖 (Mutex)：一次只允許 1 個執行緒進入臨界區。 語法糖：編譯器會自動轉換為 try…finally 包裹的 Monitor.Enter() 和 Monitor.Exit()。最簡單、最安全。 簡單的單執行緒獨佔存取共享資源（例如：更新計數器）。
Monitor 底層鎖：提供更細膩的控制能力。 功能完整：提供 Wait(), Pulse(), PulseAll() 等方法，允許執行緒在持有鎖時暫時釋放，等待特定條件後被喚醒。 需要執行緒間通訊、複雜的生產者/消費者模式。
SemaphoreSlim 信號量：一次允許 N 個執行緒進入臨界區。 計數鎖：維護一個計數器，只有計數器大於零時才允許進入。支援 async/await。 限制並行數量：限制同時處理的連線數、限制對資源的存取流量。
  1. lock： 最簡單、最安全。一次只能進一個。
  2. Monitor： lock 的底層實現，更靈活，可以控制執行緒暫停和喚醒。
  3. SemaphoreSlim： 控制最大並行數（可以同時進 N 個）。支援現代的 async/await。

請解釋 Clean Architecture 的核心概念。

核心思想是：關注點分離 (Separation of Concerns)，特別是將業務規則與外部細節完全隔離。

  1. 核心原則：依賴規則 (The Dependency Rule)
    這是 Clean Architecture 的絕對核心。
    • 規則： 程式碼的依賴關係只能由外向內，永遠不能由內向外。
    • 意義： 內層的程式碼（業務規則）對外層的程式碼（框架、資料庫、UI）一無所知。這確保了最寶貴的業務邏輯不會受到外部技術變化的影響。
  2. 架構層次 (同心圓)
    層次 (由內而外) 簡潔用途 依賴對象
    Entities (實體) 企業級業務規則。最基本的資料結構和高層次的業務規則。 任何外部事物。
    Use Cases (使用案例) 應用程式的業務規則。特定應用程式的操作流程。 僅依賴 Entities。
    Interface Adapters (介面配接器) 轉換器。負責將外部世界（如 Web, DB）的資料轉換為內層 Use Cases 能接受的格式。 依賴 Use Cases 和 Entities。
    Frameworks and Drivers (框架與驅動) 外部細節。Web 框架（ASP.NET Core）、資料庫（EF Core）、UI 界面等。 依賴 Interface Adapters。

什麼是 Memory Leak？.NET 中會怎麼發生？

記憶體洩漏是指應用程式中已經不再使用、但卻無法被釋放的記憶體空間。這些記憶體被「鎖住」或「佔用」，導致應用程式可用的記憶體越來越少，最終可能導致效能下降或應用程式崩潰。

.NET 中 Memory Leak 的常見發生方式

  • 事件訂閱未取消 (Event Subscription Leaks)
    • 當一個短生命週期的物件 (Subscriber) 訂閱了一個長生命週期的物件 (Publisher) 的事件後，Publisher 持有了 Subscriber 的強引用。
    • 必須在 Subscriber 準備被銷毀時，手動呼叫 Publisher.SomeEvent -= Handler 來取消訂閱。
  • 靜態集合持有引用 (Static Collection References)
    • 將短生命週期的物件錯誤地添加到靜態 (Static) 的 List 或 Dictionary 中。
    • 確保在使用完畢後，手動將物件從靜態集合中移除。
  • 未釋放非受管資源 (Unmanaged Resource Leaks)
    • 使用了未實現或未正確使用 IDisposable 介面的物件，或忘記使用 using 區塊來處理需要清理的資源
    • 總是對實作 IDisposable 的物件使用 using 區塊
  • 不正確的快取機制 (Caching Issues)
    • 應用程式使用了一個沒有過期策略或大小限制的快取 (Cache) 實例。
    • 確保快取系統具備合理的淘汰策略（例如：基於時間或基於容量）。
