物件導向（OOP）基礎概念

發表於 2025-11-19 分類於 基礎

物件導向(Object-oriented programming)

物件導向程式設計強調 將資料與操作資料的行為封裝成可重複使用的「零件（物件）」，再利用這些零件來組合、構建整個程式。

基本概念：類別（Class）、物件（Object）

類別（Class）

  • 類別是一種抽象的資料模型，定義事物的共同特性。
  • 包含：
    • 屬性（Field / Property）：資料的結構與內容。
    • 方法（Method）：操作資料的行為。
  • 可將「類別」視為汽車的設計藍圖，定義如：廠牌、車名、馬力等特性。

物件（Object）

  • 物件是「類別的實例（Instance）」。
  • 透過類別這張藍圖可以建立多台「汽車（物件）」，彼此獨立互不影響。

物件導向的四大特性

1. 封裝（Encapsulation）

  • 將**資料（屬性）與行為（方法）**綁在同一個物件內。
  • 隱藏內部細節，只公開必要介面供外部使用。
  • 通常透過存取修飾詞如：public、private、protected。
  • 像操作一個黑盒子，只需知道「能怎麼用」，不需理解其內部運作。

2. 繼承（Inheritance）

  • 子類別可繼承父類別的屬性與方法。
  • 建立 is-a（是一種） 的階層關係，如：
    Dog 是一種 Animal → Dog : Animal
  • 優點：
    • 程式碼重用性高。
    • 子類別可「擴展」或「覆寫（Override）」父類別方法。

3. 抽象化（Abstraction）

  • 將複雜事物簡化，只呈現必要的「介面」與「特性」。
  • 通常以 抽象類別（Abstract Class） 或 介面（Interface） 來實現。
  • 例如：開車時你只需要油門、煞車、方向盤，不需了解引擎的完整原理。

4. 多型（Polymorphism）

  • 「同一個方法介面」，可以根據物件實際型態呈現不同行為。
  • 提升程式的彈性與可擴充性。

多型的兩種形式：

  1. 方法覆寫（Method Override）
    • 子類別重新定義父類別的方法內容。
  2. 方法重載（Method Overloading）
    • 同一類別中允許相同方法名稱，但參數列表不同。

範例：

如果 Animal 類別有 makeSound() 方法：

  • Dog.makeSound() → 「汪汪」
  • Cat.makeSound() → 「喵喵」

資料庫理論筆記

發表於 2025-08-11 分類於 資料庫

1. 關聯式資料庫基本元素

  • 表格 (Relation / Table)：由列與欄組成。
  • 列 (Tuple / Row)：一筆資料。
  • 欄 (Attribute / Column)：資料屬性。
  • 主鍵 (Primary Key)：唯一識別一筆資料的欄位（或欄位組合）。
  • 外鍵 (Foreign Key)：參考另一張表主鍵的欄位，用來建立關聯。

2. 關係類型

  • 一對一 (1:1)
    例：人 ↔ 身份證
  • 一對多 (1:N)
    例：老師 ↔ 課程
  • 多對多 (M:N)
    例：學生 ↔ 課程（需用中介表實作）

3. 為什麼要正規化（Normalization）

目的：減少資料重複、避免資料異常、維持一致性。
常見異常：

  1. 重複資料（Redundancy）：同樣資訊出現多次。
  2. 更新異常（Update Anomaly）：修改時需更新多筆資料，容易漏改。
  3. 刪除異常（Deletion Anomaly）：刪除資料時，意外刪掉重要資訊。

做法：將資料拆成多張表，讓每張表專注在一種實體或關係。

4. 正規化步驟

1NF（第一正規化）

  • 定義：欄位必須是不可再分的原子值，不能有多值欄位。
  • 例子（未正規化）：
    訂單編號 客戶姓名 商品名稱 價格 數量
    001 小明 筆電, 滑鼠 35000,500 1,1
  • 轉換後（1NF）：
    訂單編號 客戶姓名 商品名稱 價格 數量
    001 小明 筆電 35000 1
    001 小明 滑鼠 500 1

2NF（第二正規化）

  • 定義：已符合 1NF，並且消除部分相依（非主鍵欄位必須完全依賴整個主鍵）。
  • 例子（未正規化）：
    訂單編號 商品名稱 客戶姓名
    001 筆電 小明
    001 滑鼠 小明

    主鍵 = (訂單編號, 商品名稱)，但「客戶姓名」只依賴「訂單編號」。

  • 轉換後（2NF）：
    • 訂單表：訂單編號、客戶姓名
    • 訂單明細表：訂單編號、商品名稱

3NF（第三正規化）

  • 定義：已符合 2NF，並且消除傳遞相依（非主鍵欄位不能依賴於其他非主鍵欄位）。
  • 例子（未正規化）：
    客戶編號 客戶姓名 客戶地址 郵遞區號
    C001 小明 台北市 100

    郵遞區號依賴地址，而地址依賴主鍵 → 間接依賴主鍵。

  • 轉換後（3NF）：
    • 客戶表：客戶編號、客戶姓名、地址
    • 郵遞區號表：地址、郵遞區號

記憶小技巧：

  • 1NF → 原子值（No 多值欄位）
  • 2NF → 消除部分依賴
  • 3NF → 消除傳遞依賴

5. 交易（Transaction）與 ACID 特性

交易（Transaction）：一組資料庫操作，要嘛全部成功，要嘛全部失敗。

ACID 特性：

  1. A - Atomicity（原子性）
    • 交易中所有操作不可分割，要嘛全部執行，要嘛全部不執行。
  2. C - Consistency（一致性）
    • 交易完成後，資料必須保持一致的狀態，符合所有規則與約束。
  3. I - Isolation（隔離性）
    • 多個交易同時進行時，彼此不應互相干擾。
  4. D - Durability（持久性）
    • 交易一旦提交，結果必須永久保留，即使系統崩潰也不能遺失。

6. 交易隔離級別（Isolation Levels）

  1. Read Uncommitted（讀未提交）
    • 可能發生：Dirty Read（髒讀）。
  2. Read Committed（讀已提交）
    • 避免髒讀，但可能有 Non-repeatable Read（不可重複讀）。
  3. Repeatable Read（可重複讀）
    • 避免不可重複讀，但可能有 Phantom Read（幻讀）。
  4. Serializable（可序列化）
    • 最高隔離，避免所有問題，但效能最低。

並行問題對照表：

隔離級別 髒讀 不可重複讀 幻讀
Read Uncommitted ✔ ✔ ✔
Read Committed ✘ ✔ ✔
Repeatable Read ✘ ✘ ✔
Serializable ✘ ✘ ✘

7. 索引（Index）

目的：加快資料檢索速度，類似書本的目錄。
原理：建立額外的資料結構（常見為 B+ Tree 或 Hash）來快速定位資料位置。

常見索引類型

  1. B+ Tree 索引
    • 適用：範圍查詢、排序查詢。
    • 優點：支援有序存取。
  2. Hash 索引
    • 適用：精確匹配查詢（=）。
    • 優點：查詢速度快。
    • 缺點：不支援範圍查詢。

索引優缺點

  • 優點：提升查詢速度、減少資料庫掃描量。
  • 缺點：
    • 占用額外儲存空間。
    • 插入、刪除、更新需要額外維護索引。

何時適合建立索引

  • 欄位經常被用於 WHERE 條件、JOIN 條件或排序（ORDER BY）。
  • 欄位值的重複度低（高選擇性）。

8. ER 模型（Entity-Relationship Model）

目的：用圖形化的方式描述資料庫中實體、屬性與關係，作為資料庫設計的第一步。

核心元素

  1. 實體（Entity）

    • 真實世界中可被識別的事物（如：學生、課程、訂單）。
    • 用長方形表示。
    • 分為：
      • 強實體（Strong Entity）：有自己的主鍵。
      • 弱實體（Weak Entity）：主鍵需依賴其他實體。

  2. 屬性（Attribute）

    • 實體的特徵（如：姓名、學號、價格）。
    • 用橢圓形表示。
    • 特殊屬性類型：
      • 複合屬性（可再拆分，如：地址 → 城市 + 郵遞區號）。
      • 多值屬性（可有多個值，如：電話號碼）。
      • 衍生屬性（可由其他屬性計算，如：年齡由生日推算）。

  3. 關係（Relationship）

    • 實體之間的聯繫（如：學生「選修」課程）。
    • 用菱形表示。
    • 分類：
      • 一對一 (1:1)
      • 一對多 (1:N)
      • 多對多 (M:N)

ER 圖符號簡記

  • 長方形：實體
  • 橢圓形：屬性
  • 菱形：關係
  • 線條：連接實體與關係，並標註基數（1、N）

ER 模型 → 關聯模型轉換原則

  1. 強實體 → 轉成一張表，屬性成為欄位，主鍵保留。
  2. 弱實體 → 轉成一張表，主鍵包含外鍵。
  3. 1:1 關係 → 可合併成一張表，或分成兩表互設外鍵。
  4. 1:N 關係 → 在 N 端加入外鍵。
  5. M:N 關係 → 建立中介表（包含兩邊的主鍵作為外鍵）。

9. 總複習小測驗

每題都對應前面的一個章節，考前自測用。

Q1（關聯式資料庫基本元素）

主鍵與外鍵的差別是什麼？請各舉一個例子。

主鍵 (Primary Key)：唯一識別一筆資料，不能重複、不能為 NULL。

例：學生表的 學號。

外鍵 (Foreign Key)：用來參考另一張表的主鍵，用於建立表之間的關聯，可以重複，可以為 NULL（視設計而定）。

例：選課表的 學號 參考學生表的 學號。

Q2（關係類型）

圖書館的「書籍」與「借閱紀錄」屬於哪種關係類型？為什麼？

「書籍」對「借閱紀錄」是 1:N，因為一本書可以對應多筆借閱紀錄，而一筆借閱紀錄只對應一本書。

Q3（為什麼要正規化）

說出至少兩個資料異常（Anomaly）的類型與例子。

1. 重複資料（Redundancy）
例：同一客戶資料在多筆訂單中重複出現。

2. 更新異常（Update Anomaly）
例：客戶搬家，要改多筆資料，漏改會導致不一致。

3. 刪除異常（Deletion Anomaly）
例：刪掉唯一一筆訂單的同時，也刪掉了該客戶的資料。

Q4（1NF~3NF）

什麼是「部分相依」？會在哪一種正規化步驟被消除？

部分相依（Partial Dependency）：
在複合主鍵的情況下，某個非主鍵欄位只依賴主鍵的一部分，而不是整個主鍵。

消除步驟：第二正規化（2NF）會移除這種依賴，確保非主鍵欄位完全依賴整個主鍵。

Q5（ACID 特性）

ACID 中的「Isolation」主要是為了解決什麼問題？

Isolation（隔離性） 的目的是：

在多個交易同時執行時，確保它們互不干擾，就好像是逐一執行一樣。

它主要是為了避免並行交易造成的問題，例如：

髒讀（Dirty Read）

不可重複讀（Non-repeatable Read）

幻讀（Phantom Read）

Q6（交易隔離級別）

哪個隔離級別可以完全避免幻讀（Phantom Read）？

Serializable（可序列化）

這是最高的隔離級別，可以完全避免髒讀、不可重複讀、幻讀。

代價是效能最低，因為它幾乎會讓交易像「排隊」一樣執行。

Q7（索引）

B+ Tree 索引和 Hash 索引各適合用在哪種查詢情境？

B+ Tree 索引 → 適合範圍查詢與排序查詢（BETWEEN、>、<、ORDER BY）

Hash 索引 → 適合精確匹配查詢（=），但不適合範圍查詢

Q8（ER 模型）

「學生」和「課程」是多對多關係，轉成關聯模型時該怎麼做？

學生表：儲存學生資訊，主鍵 學生ID

課程表：儲存課程資訊，主鍵 課程ID

中介表（選課表）：包含 學生ID 與 課程ID 作為外鍵，同時組成複合主鍵

CryptoJS 前端加密工具筆記

發表於 2025-08-01 分類於 前端

簡單記錄一下前端加密實用的工具 CryptoJS。
這是一個功能齊全的 JavaScript 加密套件，支援多種加密與雜湊演算法，使用上也非常直覺。

🔧 安裝

npm i crypto-js

若使用 TypeScript，建議額外安裝型別定義

npm i @types/crypto-js

📦 引用方式

你可以選擇引入整個套件，或是只引入你需要的模組：

// 引入整包
import CryptoJS from "crypto-js";

// 只引入 AES 模組
import AES from "crypto-js/aes";
// 只引入 SHA256 模組
import SHA256 from "crypto-js/sha256";

建議只引入需要的模組以減少打包大小。

🔐 常用功能與範例

AES 加密 / 解密

import AES from "crypto-js/aes";
import Utf8 from "crypto-js/enc-utf8";

// 使用密鑰
const secret = "my-secret-key"; // 密鑰建議使用 .env 檔案保存
// const secret = import.meta.env.VITE_CRYPTO_SECRET;
const data = "Hello, CryptoJS!";

// 加密
const encrypted = AES.encrypt(data, secret).toString();
console.log("Encrypted:", encrypted);

// 解密
const bytes = AES.decrypt(encrypted, secret);
const decrypted = bytes.toString(Utf8);
console.log("Decrypted:", decrypted);

Base64 編碼 / 解碼

import Base64 from "crypto-js/enc-base64";
import Utf8 from "crypto-js/enc-utf8";

const text = "Hello, world!";
const wordArray = Utf8.parse(text);
const encoded = Base64.stringify(wordArray);
const decoded = Base64.parse(encoded).toString(Utf8);

console.log("Encoded:", encoded);
console.log("Decoded:", decoded);

📚 支援的加密與雜湊演算法

詳情可查看List of modules

  • 雜湊：MD5、SHA1、SHA256、SHA512、RIPEMD160 等
  • 加密：AES、DES、TripleDES、Rabbit、RC4
  • 編碼：Base64、Hex、Utf8、Latin1 等

🧾 備註

🔍 適用情境

  • 加密 localStorage、cookie、URL 參數等
  • 使用 SHA256 等雜湊演算法前處理資料
  • 前端傳送加密資訊給後端（再由後端解密驗證）

❌ 不建議使用情境

  • 金鑰管理（前端無法保密金鑰）
  • 加密極機密資料（應交由後端）
  • 敏感身份驗證與權限控管

📌 參考

  • CryptoJS GitHub
  • CryptoJS 官方文件
  • CryptoJS 用前端加密、解密

DI 生命週期（LifeTime）筆記

發表於 2025-06-30 分類於 後端

在進入 LifeTime 前，我們先簡單介紹什麼是 IoC（控制反轉） 和 DI（依賴注入）

🔄 IoC（Inversion of Control）

把「誰來決定建立物件」這件事，從自己手上交給別人（框架）處理。

以前我們常會自己用 new 去建立物件：

var service = new MyService();

這種做法會導致高耦合，測試與擴充都不易。
透過 IoC，可以把物件的控制權交給「容器」處理，達到解耦與彈性注入的目的。

🎯 實際比喻

  • 傳統方式：店家收到訂單後，自行找外送員送餐
  • IoC 容器：店家透過 外送平台（IoC 容器） 處理訂單與安排外送

.NET Core 中的 DI 機制

.NET Core 內建支援 DI，使用Microsoft.Extensions.DependencyInjection 命名空間。
註冊服務時，會指定服務的生命週期（LifeTime），常見的有以下三種：

LifeTime 類型 說明 建立時機
Transient 每次請求都產生新實例 每次呼叫時
Scoped 每個 HTTP Request 共享實例 每個請求
Singleton 應用程式全域唯一實例 應用啟動時

🧬 Singleton：單例生命週期

✅ 適用情境：

  • 全域設定
  • 資料庫連線
  • 快取服務
  • 隨機數生成器、雜湊函式

🧠 使用方式：

services.AddSingleton<IMyService, MyService>();

👍 優點：

  • 只建立一次，效能佳、記憶體使用少
  • 適用於無狀態或全域狀態一致的服務

⚠️ 注意事項 :

  • 實例不能被替換
  • 不適用於每次請求需不同狀態的服務

📦 Scoped：作用域生命週期

✅ 適用情境：

  • 驗證與授權服務
  • 與 HTTP Request 綁定的服務（如使用者上下文）

🧠 使用方式：

services.AddScoped<IMyService, MyService>();

👍 優點：

  • 每個 HTTP Request 都會使用同一實例
  • 可安全操作與 Request 綁定的狀態

⚠️ 注意事項 :

  • 效能可能略低於 Singleton
  • 避免跨作用域存取（例如注入到 Singleton 裡）

✨ Transient：瞬時生命週期

✅ 適用情境：

  • 每次都要獲得新資料的服務
  • 無需維持狀態的元件

🧠 使用方式：

services.AddTransient<IMyService, MyService>();

👍 優點：

  • 每次使用都會建立新實例，資料最新
  • 不會產生跨請求污染問題

⚠️ 不建議用於 :

  • 資料庫連線、設定、快取等需要長時間共用資源的服務

SOLID

發表於 2025-06-26 分類於 後端

回顧整理複習一下基本觀念

🌟 SOLID 是什麼？

SOLID 是物件導向設計的五大原則，有助於提高系統的可讀性、可維護性與擴充性。

S – Single Responsibility Principle
O – Open/Closed Principle
L – Liskov Substitution Principle
I – Interface Segregation Principle
D – Dependency Inversion Principle

🧱 單一職責原則（SRP）

定義 : 一個模組應有且只有一個理由會使其改變。一個類別只負責一個職責(功能)。
目的 : 讓類別更專注化、模組化，便於維護與測試
範例 :

// ❌ 錯誤示範：Report 同時負責產生報告與儲存報告
public class Report {
    public void Generate() {
        // 產生報告邏輯
    }

    public void SaveToFile() {
        // 儲存報告到檔案的邏輯
    }
}

// ✅ 正確示範：拆分責任，讓每個類別只處理一件事
public class ReportGenerator {
    public Report Generate() {
        // 只負責產生報告
        return new Report();
    }
}

public class ReportSaver {
    public void SaveToFile(Report report) {
        // 只負責儲存報告
    }
}

注意不要為了拆分而拆分 => 過度拆分反而造成維護困難、結構過碎、增加開發負擔。

🔓 開放封閉原則（OCP）

定義 : 一個軟體製品應該對於擴展是開放的，但對於修改是封閉的
目的 : 當需求變更時，不用修改原有的代碼，只需擴展即可
範例 : 策略模式

// 定義一個抽象的折扣策略介面
public interface IDiscountStrategy {
    decimal GetDiscount(decimal price);
}

// 沒有折扣的策略
public class NoDiscount : IDiscountStrategy {
    public decimal GetDiscount(decimal price) => price;
}

// 季節性折扣策略
public class SeasonalDiscount : IDiscountStrategy {
    public decimal GetDiscount(decimal price) => price * 0.9m;
}

// 結帳類別依賴抽象折扣策略
public class Checkout {
    private readonly IDiscountStrategy _discountStrategy;

    public Checkout(IDiscountStrategy discountStrategy) {
        _discountStrategy = discountStrategy;
    }

    public decimal GetFinalPrice(decimal price) {
        return _discountStrategy.GetDiscount(price);
    }
}

若需新增其他折扣邏輯，如會員折扣，只需新增一個新類別實作 IDiscountStrategy，無須修改 Checkout 類別，符合 OCP。

🧬 里氏替換原則（LSP）

定義 : 子類別應該能替換其父類別，且不影響原有程式的正確性
目的 : 確保繼承的合理性，避免違反多型原則
範例 :

// 基礎類別
public class Bird {
    public virtual void Fly() { }
}

// 老鷹可以飛
public class Eagle : Bird {
    public override void Fly() { /* 飛行邏輯 */ }
}

// ❌ 鴕鳥不能飛，卻繼承 Bird，違反 LSP
public class Ostrich : Bird {
    public override void Fly() {
        throw new NotSupportedException();
    }
}

🔧 解法：可改為將「可飛的鳥」與「不可飛的鳥」抽象分開。

public abstract class Bird { }

public interface IFlyable {
    void Fly();
}

public class Eagle : Bird, IFlyable {
    public void Fly() {
        Console.WriteLine("Eagle flies.");
    }
}

public class Ostrich : Bird {
    // 沒有 Fly 方法，符合實際行為
}

🧩 介面分離原則（ISP）

定義 : 不要強迫客戶端依賴他們不使用的介面。應將大介面拆分成多個小而專一的介面。
目的 : 提高系統彈性，避免臃腫的介面影響使用。
範例 :

// ❌ 不好的設計：介面太大
public interface IMachine {
    void Print();
    void Scan();
    void Fax();
}

// 有些類別只需要部分功能
public class SimplePrinter : IMachine {
    public void Print() { /* 實作 */ }
    public void Scan() { throw new NotImplementedException(); } // 不需要
    public void Fax() { throw new NotImplementedException(); }  // 不需要
}

✅ 改善方式：拆分為小型介面

public interface IPrinter {
    void Print();
}

public interface IScanner {
    void Scan();
}

// 簡單印表機只實作印表功能
public class SimplePrinter : IPrinter {
    public void Print() {
        Console.WriteLine("Printing...");
    }
}

// 多功能印表機可以實作多個介面
public class MultiFunctionMachine : IPrinter, IScanner {
    public void Print() { /*...*/ }
    public void Scan() { /*...*/ }
}

🧲 依賴反轉原則（DIP）

定義 : 1. 高層模組不應該依賴低層模組，兩者都應該依賴抽象 2. 抽象不應依賴細節，細節應依賴抽象。
目的 : 減少模組之間的耦合，提高可重用性與可測試性。
範例 : DI


// 定義一個訊息傳送的抽象
public interface IMessageSender {
    void Send(string message);
}

// 實作細節：Email 發送
public class EmailSender : IMessageSender {
    public void Send(string message) {
        Console.WriteLine("Email sent: " + message);
    }
}

// 高層模組依賴抽象，而非 EmailSender
public class Notification {
    private readonly IMessageSender _messageSender;

    // 傳入抽象介面，實現注入（依賴反轉）
    public Notification(IMessageSender messageSender) {
        _messageSender = messageSender;
    }

    public void Notify(string message) {
        _messageSender.Send(message);
    }
}

使用如 IoC Container (例如 Microsoft.Extensions.DependencyInjection) 可以自動注入 IMessageSender 的實作。

讓ASP.NET Core服務註冊更優雅

發表於 2025-05-13 分類於 後端

前言

隨著應用程式變得越來越龐大，需要註冊的服務也隨之增加，Program.cs 中的 builder.Services 區塊可能會變得冗長且難以維護。
這時候，IServiceCollection 的擴充方法便能派上用場，幫助我們整理程式碼，使其更加清晰和易於管理。

本文範例基於 .NET 6 或更新版本（如 .NET 8）

什麼是 IServiceCollection 擴充方法？

簡單來說，擴充方法允許我們向現有的型別(IServiceCollection介面)添加新的方法，而無需修改原始型別的程式碼。
這是一個強大的 C#語言特性，可以幫助我們編寫更具表達力和可重用的程式碼。

為什麼要使用 IServiceCollection 擴充方法？

使用IServiceCollection擴充方法帶來了諸多好處

  • 提升可讀性：將特定功能的服務註冊邏輯封裝在擴充方法中，可使 Program.cs 更加簡潔，一眼看出應用程式的核心服務。
  • 增加可重用性：若多個專案或模組需註冊相同的服務集合，可將邏輯提取成擴充方法，便於重複使用。
  • 更好的組織結構：可依照模組或類型，將註冊邏輯組織到不同擴充類別中。
  • 簡化複雜的註冊邏輯：封裝需額外配置或條件註冊的服務，讓主要註冊區塊更乾淨。

如何建立和使用 IServiceCollection 擴充方法？

建立擴充方法很簡單，只需定義一個靜態類別，並撰寫靜態方法，第一個參數用 this 修飾 IServiceCollection。

1️⃣ 註冊自定義服務

假設我們有一個 MyAwesomeService 實作 IMyAwesomeService，我們希望註冊為單例：

// Extensions/ServiceCollectionExtensions.cs
using Microsoft.Extensions.DependencyInjection;

public static class ServiceCollectionExtensions
{
    public static IServiceCollection AddMyAwesomeService(this IServiceCollection services)
    {
        return services.AddSingleton<IMyAwesomeService, MyAwesomeService>();
    }
}

public interface IMyAwesomeService
{
    void DoSomethingAwesome();
}

public class MyAwesomeService : IMyAwesomeService
{
    public void DoSomethingAwesome()
    {
        Console.WriteLine("執行一些很棒的操作！");
    }
}

然後在 Program.cs 中使用

var builder = WebApplication.CreateBuilder(args);

builder.Services.AddMyAwesomeService();
// 其他服務註冊...

var app = builder.Build();
app.Run();

2️⃣ 註冊多個相關服務

我們可以一次註冊一組資料存取服務：

// Extensions/DataAccessServiceExtensions.cs
public static class DataAccessServiceExtensions
{
    public static IServiceCollection AddDataAccessServices(this IServiceCollection services, string connectionString)
    {
        services.AddScoped<IDataContext>(provider => new DataContext(connectionString));
        services.AddScoped<IUserRepository, UserRepository>();
        services.AddScoped<IProductRepository, ProductRepository>();
        return services;
    }
}

在 Program.cs 中使用：

var builder = WebApplication.CreateBuilder(args);
var connectionString = builder.Configuration.GetConnectionString("DefaultConnection");

builder.Services.AddDataAccessServices(connectionString);

var app = builder.Build();
app.Run();

3️⃣ 根據環境註冊不同的服務

根據開發或生產環境註冊不同的實作類別：

// Extensions/EnvironmentSpecificServiceExtensions.cs
using Microsoft.Extensions.Hosting;

public static class EnvironmentSpecificServiceExtensions
{
    public static IServiceCollection AddEnvironmentSpecificServices(this IServiceCollection services, IHostEnvironment environment)
    {
        if (environment.IsDevelopment())
        {
            services.AddTransient<IEmailService, DevelopmentEmailService>();
        }
        else
        {
            services.AddTransient<IEmailService, ProductionEmailService>();
        }
        return services;
    }
}

public interface IEmailService
{
    void SendEmail(string to, string subject, string body);
}

public class DevelopmentEmailService : IEmailService
{
    public void SendEmail(string to, string subject, string body)
    {
        Console.WriteLine($"[開發環境] 發送郵件至 {to}，標題：{subject}");
    }
}

public class ProductionEmailService : IEmailService
{
    public void SendEmail(string to, string subject, string body)
    {
        Console.WriteLine($"[生產環境] 正在發送郵件至 {to}，標題：{subject}...");
        // 真正的發送程式碼
    }
}

在 Program.cs 中使用：

var builder = WebApplication.CreateBuilder(args);

builder.Services.AddEnvironmentSpecificServices(builder.Environment);

var app = builder.Build();
app.Run();

結語

IServiceCollection 擴充方法是一個強大且實用的工具，能幫助我們更好地組織與管理 ASP.NET Core 中的服務註冊邏輯。
透過封裝服務註冊至擴充方法，可讓 Program.cs 更加清晰、模組化，並提升可維護性與重用性。

Dijkstra 演算法

發表於 2025-05-07 分類於 基礎

🧠 簡述

由荷蘭計算機科學家 Edsger W. Dijkstra 於 1956 年提出

  • 用來尋找單一源點到其他節點的最短路徑
  • 適用於加權圖(所以邊的權重為正)
  • 常用的資料結構 : 優先佇列(Priority Queue) 或 最小堆(Min-Heap)

🧭 步驟

  1. 將所有節點分為兩組：已確定最短路徑（S）與未確定（U）
  2. 初始時，起點在 S，其餘在 U，距離設為 ∞，起點設為 0
  3. 從 U 中選擇目前與起點距離最短的節點加入 S
  4. 更新該節點鄰接點的距離（鬆弛）
  5. 重複步驟 3~4，直到所有節點都加入 S 或無法再更新

🧑‍💻 C# 實作（含範例圖）

以下為對應圖形的 C# 程式實作

static void Main()
{
    int n = 7; // 節點數
    List<Edge>[] graph = new List<Edge>[n];
    for (int i = 0; i < n; i++) graph[i] = new List<Edge>();

    // 建立圖的鄰接表（根據圖像）
    AddEdge(graph, 0, 1, 2);
    AddEdge(graph, 0, 2, 6);
    AddEdge(graph, 1, 3, 5);
    AddEdge(graph, 2, 3, 8);
    AddEdge(graph, 3, 4, 10);
    AddEdge(graph, 3, 5, 15);
    AddEdge(graph, 4, 5, 6);
    AddEdge(graph, 4, 6, 2);
    AddEdge(graph, 5, 6, 6);

    int start = 0;
    int[] dist = Dijkstra(graph, start);

    Console.WriteLine("起點到各節點的最短距離");
    for (int i = 0; i < n; i++)
    {
        Console.WriteLine($"到節點{i}的最短距離 : {dist[i]}");
    }
}

class Edge
{
    public int To, Weight;
    public Edge(int to, int weight)
    {
        To = to; Weight = weight;
    }
}

static void AddEdge(List<Edge>[] graph, int from, int to, int weight)
{
    graph[from].Add(new Edge(to, weight));
    graph[to].Add(new Edge(from, weight)); // 無向圖需要雙向邊
}

static int[] Dijkstra(List<Edge>[] graph, int start)
{
    int n = graph.Length;
    int[] dist = new int[n];
    bool[] visited = new bool[n];

    // 初始化距離與訪問狀態
    for (int i = 0; i < n; i++) dist[i] = int.MaxValue;
    dist[start] = 0;

    // 使用NET6+ 的 PriorityQueue<TElement, TPriority>
    // 舊版要自己實作 MinHeap
    // 使用優先佇列處理節點，依距離最小優先處理
    var pq = new PriorityQueue<int, int>();
    pq.Enqueue(start, 0);

    while (pq.Count > 0)
    {
        int u = pq.Dequeue();

        // 確認是否處理過節點，並更新註記
        if (visited[u]) continue;
        visited[u] = true;

        // 遍歷所有相鄰邊
        foreach (var edge in graph[u])
        {
            int v = edge.To;
            int weight = edge.Weight;

           // 若從 u 到 v 的新距離更短，則更新 dist 並放入優先佇列
            if (dist[u] + weight < dist[v])
            {
                dist[v] = dist[u] + weight;
                pq.Enqueue(v, dist[v]);
            }
        }
    }

    return dist;
}

WebRTC

發表於 2025-05-01 更新於 2025-09-15 分類於 後端

簡易記錄一下 WebRTC 的基本概念

基本建立點對點連線的流程

信令 (Signaling)：

  1. 兩個 WebRTC 客户端（Peer A 和 Peer B）透過一個信令伺服器進行初始的資訊交換。這個伺服器不參與實際的媒體傳輸。
  2. Peer A (發起方 - Offerer) 創建一個包含其媒體能力和網路資訊的 SDP Offer。
  3. Peer A 收集其 ICE Candidates（潛在的網路位址）。
  4. Peer A 將 SDP Offer 和 ICE Candidates 透過信令伺服器發送給 Peer B。
  5. Peer B (接收方 - Answerer) 收到 SDP Offer 和 ICE Candidates。
  6. Peer B 設定遠端的 SDP Offer，並創建包含其自身媒體能力和網路資訊的 SDP Answer。
  7. Peer B 也收集其 ICE Candidates。
  8. Peer B 將 SDP Answer 和 ICE Candidates 透過信令伺服器發送回 Peer A。
  9. Peer A 收到 SDP Answer 和 ICE Candidates，並設定遠端的 SDP Answer。
  10. 雙方持續透過信令伺服器交換 ICE Candidates。

ICE 連線建立 (ICE Connection Establishment)：

  1. 每個 Peer 在收到對方的 ICE Candidates 後，會嘗試與這些候選者建立網路連線。
  2. 這個過程會嘗試各種可能的路徑（直接連接、透過 STUN、透過 TURN）。
  3. 當找到一個可用的連線路徑時，ICE 連線建立成功。

媒體和資料傳輸：

  1. 一旦 ICE 連線建立，兩個 Peer 就可以直接透過這個點對點連線傳輸音訊、視訊或任意資料（使用 Data Channels）。

相關名詞簡介：

WebRTC (Web Real-Time Communication)：

一個開放原始碼專案，提供瀏覽器和行動應用程式之間即時音訊、視訊和資料交換的能力，無需外掛程式。(目前只有 js)

Peer (對等方)：

在 WebRTC 通訊中的一個端點，通常是一個瀏覽器或應用程式。

點對點 (Peer-to-Peer, P2P)：

一種網路模型，其中兩個或多個參與者可以直接互相通信，而不需要透過中央伺服器（在媒體和資料傳輸階段）。

信令 (Signaling)：

在 WebRTC 連線建立之前，兩個 Peer 之間交換控制資訊（例如 SDP 和 ICE Candidates）的過程。信令機制不是 WebRTC 本身定義的，需要開發者自行實作。

信令伺服器 (Signaling Server)：

一個中介伺服器，用於在 WebRTC 連線建立的初始階段協助 Peer 發現彼此並交換信令訊息。

SDP (Session Description Protocol)：

一種標準化的文字格式，用於描述多媒體會話的屬性，例如支援的媒體類型、編解碼器、網路位址等。在 WebRTC 中，SDP 用於在連接建立前協商媒體格式。

Offer/Answer 模型 (Offer/Answer Model)：

WebRTC 使用這個模型來協商會話參數。發起方創建一個 Offer SDP，接收方回應一個 Answer SDP。

ICE (Interactive Connectivity Establishment)：

一個用於 NAT 穿透的協定。ICE 的目標是找到兩個 Peer 之間最佳的網路路徑來建立直接連線。

ICE Candidate (ICE 候選者)：

描述一個 Peer 可能的網路位址和連接埠的資訊，用於 ICE 連線建立過程。

STUN (Session Traversal Utilities for NAT) 伺服器：

一種協助位於 NAT 後面的 Peer 發現其公有 IP 位址和連接埠的伺服器。

TURN (Traversal Using Relays around NAT) 伺服器：

一種在直接連線和 STUN 無法工作時，作為中繼伺服器轉發 Peer 之間流量的伺服器。

NAT (Network Address Translator)：

一種網路設備，用於將私有網路中的多個裝置的 IP 位址轉換為一個或少數幾個公有 IP 位址，以便它們可以存取網際網路。NAT 會導致直接的點對點連線變得困難。
