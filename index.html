🦀 Rust 基礎教學：變數、常數與遮蔽（Shadowing）

發表於 2025-05-13 分類於 rust

📌 變數的宣告

Rust 使用 let 來宣告變數，預設為不可變（immutable）。如果想讓變數的值可以改變，需要加上 mut。

let a = 456;       // 不可變變數
let mut b = 789;   // 可變變數
println!("a = {}, b = {}", a, b);

// a = 999; // ❌ 錯誤：a 是不可變的
b = 111;   // ✅ 合法：b 是可變的
println!("a = {}, b = {}", a, b);

🔒 常數（const）

常數使用 const 宣告，與 let 不同的是：

  • 必須加上資料別
  • 永遠不可變(不能加 mut)
  • 通常用於全域設定或固定值
  • 命名習慣 SCREAMING_SNAKE_CASE
const MAX_POINTS: u32 = 100_000;
println!("The maximum points are: {}", MAX_POINTS);

⚠️ 常數不能遮蔽（shadow），也不能重新指定

🎭 變數遮蔽（Shadowing）

Rust 允許重新宣告相同名稱的變數，這稱為遮蔽（shadowing）。
遮蔽可以讓你重新指定值甚至改變變數型別，而不用使用 mut。

let c = 10;
let c = "abc"; // 遮蔽了上一個 c，型別從 i32 → &str
println!("The value of c is: {}", c);

與mut有何不同?

let mut x = 5;
x = "hello"; // ❌ 錯誤：型別不能變

let y = 5;
let y = "hello"; // ✅ 合法：因為是新的變數，遮蔽前一個

✅ 使用遮蔽能讓變數在不改變可變性的情況下重新定義或轉換型別。

📝 小結

宣告方式 是否可變 是否可遮蔽 是否需要型別註記 命名風格
let ❌ 預設不可變（可加 mut） ✅ 可以遮蔽 ❌ 可省略 snake_case
const ❌ 永遠不可變 ❌ 不可遮蔽 ✅ 必須註明型別 SCREAMING_SNAKE_CASE

讓ASP.NET Core服務註冊更優雅

發表於 2025-05-13 分類於 後端

前言

隨著應用程式變得越來越龐大，需要註冊的服務也隨之增加，Program.cs 中的 builder.Services 區塊可能會變得冗長且難以維護。
這時候，IServiceCollection 的擴充方法便能派上用場，幫助我們整理程式碼，使其更加清晰和易於管理。

本文範例基於 .NET 6 或更新版本（如 .NET 8）

什麼是 IServiceCollection 擴充方法？

簡單來說，擴充方法允許我們向現有的型別(IServiceCollection介面)添加新的方法，而無需修改原始型別的程式碼。
這是一個強大的 C#語言特性，可以幫助我們編寫更具表達力和可重用的程式碼。

為什麼要使用 IServiceCollection 擴充方法？

使用IServiceCollection擴充方法帶來了諸多好處

  • 提升可讀性：將特定功能的服務註冊邏輯封裝在擴充方法中，可使 Program.cs 更加簡潔，一眼看出應用程式的核心服務。
  • 增加可重用性：若多個專案或模組需註冊相同的服務集合，可將邏輯提取成擴充方法，便於重複使用。
  • 更好的組織結構：可依照模組或類型，將註冊邏輯組織到不同擴充類別中。
  • 簡化複雜的註冊邏輯：封裝需額外配置或條件註冊的服務，讓主要註冊區塊更乾淨。

如何建立和使用 IServiceCollection 擴充方法？

建立擴充方法很簡單，只需定義一個靜態類別，並撰寫靜態方法，第一個參數用 this 修飾 IServiceCollection。

1️⃣ 註冊自定義服務

假設我們有一個 MyAwesomeService 實作 IMyAwesomeService，我們希望註冊為單例：

// Extensions/ServiceCollectionExtensions.cs
using Microsoft.Extensions.DependencyInjection;

public static class ServiceCollectionExtensions
{
    public static IServiceCollection AddMyAwesomeService(this IServiceCollection services)
    {
        return services.AddSingleton<IMyAwesomeService, MyAwesomeService>();
    }
}

public interface IMyAwesomeService
{
    void DoSomethingAwesome();
}

public class MyAwesomeService : IMyAwesomeService
{
    public void DoSomethingAwesome()
    {
        Console.WriteLine("執行一些很棒的操作！");
    }
}

然後在 Program.cs 中使用

var builder = WebApplication.CreateBuilder(args);

builder.Services.AddMyAwesomeService();
// 其他服務註冊...

var app = builder.Build();
app.Run();

2️⃣ 註冊多個相關服務

我們可以一次註冊一組資料存取服務：

// Extensions/DataAccessServiceExtensions.cs
public static class DataAccessServiceExtensions
{
    public static IServiceCollection AddDataAccessServices(this IServiceCollection services, string connectionString)
    {
        services.AddScoped<IDataContext>(provider => new DataContext(connectionString));
        services.AddScoped<IUserRepository, UserRepository>();
        services.AddScoped<IProductRepository, ProductRepository>();
        return services;
    }
}

在 Program.cs 中使用：

var builder = WebApplication.CreateBuilder(args);
var connectionString = builder.Configuration.GetConnectionString("DefaultConnection");

builder.Services.AddDataAccessServices(connectionString);

var app = builder.Build();
app.Run();

3️⃣ 根據環境註冊不同的服務

根據開發或生產環境註冊不同的實作類別：

// Extensions/EnvironmentSpecificServiceExtensions.cs
using Microsoft.Extensions.Hosting;

public static class EnvironmentSpecificServiceExtensions
{
    public static IServiceCollection AddEnvironmentSpecificServices(this IServiceCollection services, IHostEnvironment environment)
    {
        if (environment.IsDevelopment())
        {
            services.AddTransient<IEmailService, DevelopmentEmailService>();
        }
        else
        {
            services.AddTransient<IEmailService, ProductionEmailService>();
        }
        return services;
    }
}

public interface IEmailService
{
    void SendEmail(string to, string subject, string body);
}

public class DevelopmentEmailService : IEmailService
{
    public void SendEmail(string to, string subject, string body)
    {
        Console.WriteLine($"[開發環境] 發送郵件至 {to}，標題：{subject}");
    }
}

public class ProductionEmailService : IEmailService
{
    public void SendEmail(string to, string subject, string body)
    {
        Console.WriteLine($"[生產環境] 正在發送郵件至 {to}，標題：{subject}...");
        // 真正的發送程式碼
    }
}

在 Program.cs 中使用：

var builder = WebApplication.CreateBuilder(args);

builder.Services.AddEnvironmentSpecificServices(builder.Environment);

var app = builder.Build();
app.Run();

結語

IServiceCollection 擴充方法是一個強大且實用的工具，能幫助我們更好地組織與管理 ASP.NET Core 中的服務註冊邏輯。
透過封裝服務註冊至擴充方法，可讓 Program.cs 更加清晰、模組化，並提升可維護性與重用性。

Dijkstra 演算法

發表於 2025-05-07 分類於 基礎

🧠 簡述

由荷蘭計算機科學家 Edsger W. Dijkstra 於 1956 年提出

  • 用來尋找單一源點到其他節點的最短路徑
  • 適用於加權圖(所以邊的權重為正)
  • 常用的資料結構 : 優先佇列(Priority Queue) 或 最小堆(Min-Heap)

🧭 步驟

  1. 將所有節點分為兩組：已確定最短路徑（S）與未確定（U）
  2. 初始時，起點在 S，其餘在 U，距離設為 ∞，起點設為 0
  3. 從 U 中選擇目前與起點距離最短的節點加入 S
  4. 更新該節點鄰接點的距離（鬆弛）
  5. 重複步驟 3~4，直到所有節點都加入 S 或無法再更新

🧑‍💻 C# 實作（含範例圖）

以下為對應圖形的 C# 程式實作

static void Main()
{
    int n = 7; // 節點數
    List<Edge>[] graph = new List<Edge>[n];
    for (int i = 0; i < n; i++) graph[i] = new List<Edge>();

    // 建立圖的鄰接表（根據圖像）
    AddEdge(graph, 0, 1, 2);
    AddEdge(graph, 0, 2, 6);
    AddEdge(graph, 1, 3, 5);
    AddEdge(graph, 2, 3, 8);
    AddEdge(graph, 3, 4, 10);
    AddEdge(graph, 3, 5, 15);
    AddEdge(graph, 4, 5, 6);
    AddEdge(graph, 4, 6, 2);
    AddEdge(graph, 5, 6, 6);

    int start = 0;
    int[] dist = Dijkstra(graph, start);

    Console.WriteLine("起點到各節點的最短距離");
    for (int i = 0; i < n; i++)
    {
        Console.WriteLine($"到節點{i}的最短距離 : {dist[i]}");
    }
}

class Edge
{
    public int To, Weight;
    public Edge(int to, int weight)
    {
        To = to; Weight = weight;
    }
}

static void AddEdge(List<Edge>[] graph, int from, int to, int weight)
{
    graph[from].Add(new Edge(to, weight));
    graph[to].Add(new Edge(from, weight)); // 無向圖需要雙向邊
}

static int[] Dijkstra(List<Edge>[] graph, int start)
{
    int n = graph.Length;
    int[] dist = new int[n];
    bool[] visited = new bool[n];

    // 初始化距離與訪問狀態
    for (int i = 0; i < n; i++) dist[i] = int.MaxValue;
    dist[start] = 0;

    // 使用NET6+ 的 PriorityQueue<TElement, TPriority>
    // 舊版要自己實作 MinHeap
    // 使用優先佇列處理節點，依距離最小優先處理
    var pq = new PriorityQueue<int, int>();
    pq.Enqueue(start, 0);

    while (pq.Count > 0)
    {
        int u = pq.Dequeue();

        // 確認是否處理過節點，並更新註記
        if (visited[u]) continue;
        visited[u] = true;

        // 遍歷所有相鄰邊
        foreach (var edge in graph[u])
        {
            int v = edge.To;
            int weight = edge.Weight;

           // 若從 u 到 v 的新距離更短，則更新 dist 並放入優先佇列
            if (dist[u] + weight < dist[v])
            {
                dist[v] = dist[u] + weight;
                pq.Enqueue(v, dist[v]);
            }
        }
    }

    return dist;
}

WebRTC

發表於 2025-05-01 分類於 後端

簡易記錄一下 WebRTC 的基本概念

基本建立點對點連線的流程

信令 (Signaling)：

  1. 兩個 WebRTC 客户端（Peer A 和 Peer B）透過一個信令伺服器進行初始的資訊交換。這個伺服器不參與實際的媒體傳輸。
  2. Peer A (發起方 - Offerer) 創建一個包含其媒體能力和網路資訊的 SDP Offer。
  3. Peer A 收集其 ICE Candidates（潛在的網路位址）。
  4. Peer A 將 SDP Offer 和 ICE Candidates 透過信令伺服器發送給 Peer B。
  5. Peer B (接收方 - Answerer) 收到 SDP Offer 和 ICE Candidates。
  6. Peer B 設定遠端的 SDP Offer，並創建包含其自身媒體能力和網路資訊的 SDP Answer。
  7. Peer B 也收集其 ICE Candidates。
  8. Peer B 將 SDP Answer 和 ICE Candidates 透過信令伺服器發送回 Peer A。
  9. Peer A 收到 SDP Answer 和 ICE Candidates，並設定遠端的 SDP Answer。
  10. 雙方持續透過信令伺服器交換 ICE Candidates。

ICE 連線建立 (ICE Connection Establishment)：

  1. 每個 Peer 在收到對方的 ICE Candidates 後，會嘗試與這些候選者建立網路連線。
  2. 這個過程會嘗試各種可能的路徑（直接連接、透過 STUN、透過 TURN）。
  3. 當找到一個可用的連線路徑時，ICE 連線建立成功。

媒體和資料傳輸：

  1. 一旦 ICE 連線建立，兩個 Peer 就可以直接透過這個點對點連線傳輸音訊、視訊或任意資料（使用 Data Channels）。

相關名詞簡介：

WebRTC (Web Real-Time Communication)：

一個開放原始碼專案，提供瀏覽器和行動應用程式之間即時音訊、視訊和資料交換的能力，無需外掛程式。(目前只有 js)

Peer (對等方)：

在 WebRTC 通訊中的一個端點，通常是一個瀏覽器或應用程式。

點對點 (Peer-to-Peer, P2P)：

一種網路模型，其中兩個或多個參與者可以直接互相通信，而不需要透過中央伺服器（在媒體和資料傳輸階段）。

信令 (Signaling)：

在 WebRTC 連線建立之前，兩個 Peer 之間交換控制資訊（例如 SDP 和 ICE Candidates）的過程。信令機制不是 WebRTC 本身定義的，需要開發者自行實作。

信令伺服器 (Signaling Server)：

一個中介伺服器，用於在 WebRTC 連線建立的初始階段協助 Peer 發現彼此並交換信令訊息。

SDP (Session Description Protocol)：

一種標準化的文字格式，用於描述多媒體會話的屬性，例如支援的媒體類型、編解碼器、網路位址等。在 WebRTC 中，SDP 用於在連接建立前協商媒體格式。

Offer/Answer 模型 (Offer/Answer Model)：

WebRTC 使用這個模型來協商會話參數。發起方創建一個 Offer SDP，接收方回應一個 Answer SDP。

ICE (Interactive Connectivity Establishment)：

一個用於 NAT 穿透的協定。ICE 的目標是找到兩個 Peer 之間最佳的網路路徑來建立直接連線。

ICE Candidate (ICE 候選者)：

描述一個 Peer 可能的網路位址和連接埠的資訊，用於 ICE 連線建立過程。

STUN (Session Traversal Utilities for NAT) 伺服器：

一種協助位於 NAT 後面的 Peer 發現其公有 IP 位址和連接埠的伺服器。

TURN (Traversal Using Relays around NAT) 伺服器：

一種在直接連線和 STUN 無法工作時，作為中繼伺服器轉發 Peer 之間流量的伺服器。

NAT (Network Address Translator)：

一種網路設備，用於將私有網路中的多個裝置的 IP 位址轉換為一個或少數幾個公有 IP 位址，以便它們可以存取網際網路。NAT 會導致直接的點對點連線變得困難。

進階

就算設定環境變數來避免將敏感資料寫入程式，比 appsettings.json 安全，但是還是有相關風險
所以可能會進一步需要機密管理服務（Secrets Management Services）來去管理敏感資訊
但這邊用不到就大概紀錄一下可能的更優解，因為要 💸

環境變數的侷限

問題點 說明
被誤讀 足夠權限的使用者可透過指令查看所有變數
難管理 每台機器都要單獨設定，不適合大型或多環境部署
無審計紀錄 無法追蹤誰改了哪個變數、何時改的
版本控管 無法追蹤歷史紀錄或自動回復舊版本
變更難通知 改變值，需要人為通知開發 / 系統人員重啟服務

Azure Key Vault 為例的優勢

功能 好處
集中管理 所有機密集中存於一個 Vault，減少散落風險
權限控管 使用 Azure AD、RBAC 控制誰能取用哪些機密
自動輪換密鑰 可結合自動化工具定期更新連線密碼、憑證等
加密儲存 Vault 本身以高強度加密保存內容
稽核與日誌 每次存取都會記錄，便於日後查詢與調查
不用再碰密碼 程式碼用 Identity 存取 Vault，不需要明碼存在任何地方

前端生成PDF

發表於 2025-04-25 分類於 前端

前言

因為業務需求的關係，需要將填寫的表單轉換成 PDF，進行資料儲存供後續使用
最便捷的方法為是使用前端套件html2canvas搭配jspdf
利用html2canvas進行截圖，然後貼到jspdf產生的 PDF 檔
但最穩妥的方法建議是使用後端服務器去專門產生 PDF 比較合適(ex. wkhtmltopdf 或是 Puppeteer )

Install

npm install html2canvas jspdf

Usage

pdfHelper.ts

import jsPDF from "jspdf";
import html2canvas from "html2canvas";

export const generatePDF = async (id: string) => {
  try {
    // 取得要擷取的部分
    const el = document.getElementById(id) as HTMLElement;
    if (!el) return "";

    // 截圖
    const canvas = await html2canvas(el, {
      useCORS: true, // 允許跨域圖片
      scale: 2, // 提高清晰度
    });

    // 轉換圖片
    // toDataURL({轉換格式}, {品質})
    const imgData = canvas.toDataURL("image/jpeg", 0.7);

    // 創建PDF文檔
    const pdf = new jsPDF({
      orientation: "portrait", // 頁面方向
      unit: "pt", // 單位
      format: "a4", // 紙張大小
    });

    // 取得PDF尺寸
    const pageWidth = pdf.internal.pageSize.getWidth();
    const pageHeight = pdf.internal.pageSize.getHeight();

    // 調整為A4寬度且維持原本寬高比
    const imgWidth = pageWidth;
    const imgHeight = (canvas.height * imgWidth) / canvas.width;

    // 若圖像高度超出頁面高度，則縮小
    const finalHeight = imgHeight > pageHeight ? pageHeight : imgHeight;
    const finalWidth = (canvas.width * finalHeight) / canvas.height;

    // 可選：讓圖片置中
    const marginX = (pageWidth - finalWidth) / 2;
    const marginY = (pageHeight - finalHeight) / 2;

    // 添加圖片到PDF
    pdf.addImage(
      imgData, // 圖片資料
      "JPEG", // 圖片格式
      marginX, // X座標
      marginY, // Y座標
      finalWidth, //圖片寬度
      finalHeight, //圖片高度
      undefined, // 別名
      "FAST" // 壓縮方法
    );

    // 取得包含 Base64 的 Data URL
    const pdfDataUrl = pdf.output("datauristring");

    // Data URL 的格式通常是 "data:application/pdf;base64,BASE64_STRING"
    // 我們需要提取逗號後面的 Base64 部分
    const base64String = pdfDataUrl.split(",")[1];

    return base64String;
  } catch (e) {
    console.log(e);
    throw e;
  }
};

簡介

SqlKata 查詢生成器是一個用 C#編寫的功能強大 SQL 生成器，它安全、不受框架影響，靈感來源於 Laravel Query Builder 和 Knex 等頂級查詢建立工具

  • 乾淨的命名，與 SQL 語法相似
  • 允許統一 API 處理多個數據庫，且可搭配 Dapper 使用，有效替代數據庫的存儲過程
  • 支持複雜查詢，如:嵌套條件、子查詢…等
  • 目前有支援的資料庫 SqlServer、MySql、PostgreSql、Oracle …等
  • 擴充項目，搭配 EntityFramework Core 6 使用
  • 詳細文檔
  • PlayGround

安裝

$ dotnet add package SqlKata
$ dotnet add package SqlKata.Execution # (選用) 如果您希望執行支援

範例

簡易實作(以 Oracle 為例)

var connection = new OracleConnection(connectionString);
var compiler = new OracleCompiler();

Dependency injection 注入實作(For .NET 6)

QueryFactoryFactory.cs

public class QueryFactoryFactory : IQueryFactoryFactory
{
    private readonly IConfiguration _configuration;
    public QueryFactoryFactory(IConfiguration configuration)
    {
        _configuration = configuration;
    }
    public QueryFactory CreateQueryFactory(string dbKey)
    {
        var cs = _configuration.GetConnectionString(dbKey);
        if (string.IsNullOrEmpty(cs))
        {
            throw new ArgumentException($"Connection string for {dbKey} is not found.");
        }
        var conn = new OracleConnection(cs);
        var compiler = new OracleCompiler();
        return new QueryFactory(conn, compiler);
    }
}

public interface IQueryFactoryFactory
{
    QueryFactory CreateQueryFactory(string dbKey);
}

Program.cs

builder.Services.AddTransient<IQueryFactoryFactory, QueryFactoryFactory>();

HomeController.cs

using SqlKata;
using SqlKata.Execution;

public class HomeController {

    private readonly IQueryFactoryFactory factory;

    public HomeController(IQueryFactoryFactory factory)
    {
       this.factory = factory;
    }

    public async Task<IActionResult> Get()
    {
       using var db = factory.CreateQueryFactory(dbKey);

       // 這裡 books 的類型是 `IEnumerable<dynamic>`
       var books = await db.Query("Books")
           .Where("IsPublished", true)
           .GetAsync();

       // or `IEnumerable<Book>` if using the Generic overload
       var books = await db.Query("Books")
           .Where("IsPublished", true)
           .GetAsync<Book>();

       return Ok(books);
    }
}

BIOS

基本輸入輸出系統(Basic Input/Output System)

主要作用為將系統硬體帶到可以準備交給作業系統的狀態，通常包含下列步驟:

  • 系統初始化
  • 測試記憶體與其他周邊
  • 檢測儲存裝置
  • 在數個儲存裝置之間找出作業系統的起始點

且提供操作介面，供使用者設定系統的低階參數、選擇開機裝置、配置系統資源等等。

日期與時間

BIOS 的主要功能之一是管理主機板內建的硬體時鐘。
作業系統本身也有設定硬體時鐘的能力，可以依靠精準的外部較時參考來源(ex: NTPD 伺服器)來保持硬體時鐘的精準。

磁碟與開機裝置

BIOS 設定工具中其中一項重要功能就是開機裝置的選擇。
大多數 PC 至少支援三種開機媒體

  • 內接式硬碟(IDE 或 SCSI 或 SATA，或三者皆是)
  • 光碟機(同上)
  • 軟碟機

運用/proc 檔案系統

/proc 檔案系統 (核心的運作狀態庫)目錄下的 interrupts、dma、ioports 三個檔案，提供目前系統資源的使用狀況

USB

USB 拓樸邏輯

電腦主機與 USB 裝置之間，是透過集線器(HUB)連接一起，構成一個樹狀結構。
使用lsusb指令可顯示 Linux 系統上所有 USB 裝置的實際連接方式。

USB 控制器

主要分成 4 種類型

  • 開放型主控界面 ( Open Host Controller Interface , OHCI )
  • 通用型主控界面 ( Universal Host Controller Interface , UJCI )
  • 強化型主控界面 ( Enhanced Host Controller Interface , EHCI )
  • 擴充型主控界面 ( eXtented Host Controller Interface , xHCI )
規格 控制器 傳輸速率 向下相容
USB1.1 OHCI、UHCI 12Mbps 否
USB2.0 EHCI 480Mbps 否
USB3.0 xHCI 5Gbps 是

USB 裝置

  • 人機界面裝置 ( Human Interface Device , HID)
    輸入裝置 (滑鼠、鍵盤…等等)
  • 通訊裝置
    數據機
  • 大量儲存裝置
    磁碟機、讀卡機…等
  • 音訊
    揚聲器、音效裝置
  • IrDA
    紅外線裝置
  • 印表機
    印表機以及 USB-to-parallel 轉換器

USB 驅動程式

Linux 核心 2.0.x 以及更早版本，不支援 USB

主要分成三大類

  1. 主控器驅動程式
  2. 通類驅動程式
  3. 雜項裝置驅動程式

樹狀資料

發表於 2025-02-05 更新於 2025-04-25 分類於 基礎

樹狀結構介紹

  • Node : 樹的基本單位，紀錄節點資料與子節點有哪些
  • Root Node : 根節點，樹的最高節點
  • Leaf Node : 葉節點，沒有子節點的節點
  • Edge : 連接兩個節點的線，表示節點之間的關係
  • Parent Node : 父節點(上層節點)，一個節點只會有一個父節點
  • Child Node : 子節點，一個節點可以連結多個子節點
  • Sibling Node : 兄弟節點，有相同父節點的節點
  • Depth : 從根節點到該節點的路徑長度
  • Height : 該節點到最遠葉節點的路徑長度
  • Level : 節點層次
  • Subtree : 某節點的所有子節點後代

樹的種類

  1. 二元樹 (Binary Tree)
    • 一個節點最多兩個子節點
  2. 二元搜尋樹 (Binary Search Tree, BST)
    • 左子樹所有節點值小於根節點值
    • 右子樹所有節點值大於根節點值
  3. 平衡樹 (Balanced Tree)
  4. 滿二元樹 (Full Binary Tree)
  5. 完全二元樹 (Complete Binary Tree)
  6. B 樹 (B-Tree)
  7. 堆積 (Heap)
  8. 字典樹 (Trie)

樹的遍歷(Traversal)

  1. 深度優先遍歷 (Depth-First Search, DFS)
    • 先訪問子節點，再訪問兄弟節點
    • 3 種遍歷方法
      • 前序遍歷 (Pre-order)
      • 中序遍歷 (In-order)
      • 後續遍歷 (Post-order)
  2. 廣度優先遍歷 (Breadth-First Search, BFS)
    • 階層遍歷，一層一層從左到右

相關實作(C#)

// 節點
public class TreeNode
{
    public int val;
    public TreeNode left;
    public TreeNode right;
    public TreeNode(int val = 0, TreeNode left = null, TreeNode right = null)
    {
        this.val = val;
        this.left = left;
        this.right = right;
    }
}

// 深度優先遍歷 (Depth-First Search, DFS)
// 前序遍歷
public void PreorderTraversal(TreeNode root)
{
    if (root == null) return;
    Console.Write(root.val + " ") // 訪問節點
    PreorderTraversal(root.left); // 遞迴訪問左子樹
    PreorderTraversal(root.right); // 遞迴訪問右子樹
}

// 中序遍歷
public void InorderTraversal(TreeNode root)
{
    if (root == null) return;
    InorderTraversal(root.left);    // 遞迴遍歷左子樹
    Console.Write(root.val + " ");  // 訪問節點
    InorderTraversal(root.right);   // 遞迴遍歷右子樹
}

// 後序遍歷
public void PostorderTraversal(TreeNode root)
{
    if (root == null) return;
    PostorderTraversal(root.left);   // 遞迴遍歷左子樹
    PostorderTraversal(root.right);  // 遞迴遍歷右子樹
    Console.Write(root.val + " ");   // 訪問節點
}

// 廣度優先遍歷 (Breadth-First Search, BFS)
// 階層遍歷
public void LevelOrderTraversal(TreeNode root)
{
    if (root == null) return;

    Queue<TreeNode> queue = new Queue<TreeNode>();
    queue.Enqueue(root);

    while (queue.Count > 0)
    {
        TreeNode node = queue.Dequeue();
        Console.Write(node.val + " ");

        if (node.left != null) queue.Enqueue(node.left);
        if (node.right != null) queue.Enqueue(node.right);
    }
}
